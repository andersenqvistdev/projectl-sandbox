# /company-hire — Hire a Company Employee

Hire a new employee with specific skills. The employee is generated by the Meta-Agent, assigned to a department, and given working memory. Employees are company-level resources that can be assigned to projects.

## Input
$ARGUMENTS

Expected format: `[skill/role description] [--department=X] [--team=Y] [--type=persistent|consultant]`

Examples:
- `performance optimization specialist`
- `database migration expert --department=engineering --team=devops`
- `user research analyst --department=product`
- `security auditor --type=consultant`

## Step 0: Determine Operating Mode

Use `company_resolver` to determine if operating in multi-project mode:

```bash
uv run .claude/hooks/company/company_resolver.py mode 2>/dev/null && echo "MULTI_PROJECT" || echo "SINGLE_PROJECT"
```

Also get the company directory path:

```bash
uv run .claude/hooks/company/company_resolver.py dir 2>/dev/null
```

**Store the results:**
- `$OPERATING_MODE`: "MULTI_PROJECT" or "SINGLE_PROJECT"
- `$COMPANY_DIR`: Path to `.company/` directory (resolved to company root in multi-project mode)

## Step 0.1: Validate Company Structure

Check that company is initialized:

```bash
ls $COMPANY_DIR/org.json 2>/dev/null || echo "NOT_INITIALIZED"
```

**If NOT_INITIALIZED:**

For multi-project mode:
```
## Company Not Initialized

No company structure found.

**Multi-project mode detected** — Run `/company-create` to create a company root,
or `/company-init` to create a single-project company in the current directory.
```

For single-project mode:
```
## Company Not Initialized

The company directory structure does not exist.

Run `/company-init` first to create the organizational structure.
```

Exit without changes.

## Step 1: Parse Arguments

Parse `$ARGUMENTS` to extract:

1. **Role Description** (required): The skill/role description (everything before `--` flags)
2. **Department** (optional): `--department=X` flag (default: auto-detect from role)
3. **Team** (optional): `--team=Y` flag (default: null, assigned later)
4. **Type** (optional): `--type=persistent|consultant` (default: "persistent")

**If no role description provided:**
```
## Missing Role Description

Usage: /company-hire [role description] [--department=X] [--team=Y] [--type=TYPE]

Examples:
  /company-hire performance optimization specialist
  /company-hire database migration expert --department=engineering
  /company-hire user research analyst --department=product --team=user-research
  /company-hire security auditor --type=consultant
```

Exit without changes.

## Step 2: Auto-Detect Department (if not specified)

If no `--department` flag, analyze the role description to determine the best fit:

| Keywords in Role | Department |
|------------------|------------|
| engineer, developer, code, backend, frontend, api, database, devops, infrastructure, performance, security, testing | engineering |
| product, manager, strategy, roadmap, requirements, prioritization | product |
| design, ux, ui, visual, user experience, interface, prototype, wireframe | design |

**If ambiguous:** Default to `engineering` (most common case).

## Step 3: Generate Agent ID

Create a unique agent ID from the role description:

1. Extract key words from role description
2. Convert to lowercase, hyphenated format
3. Add random suffix if ID already exists in org.json

Example: "Performance Optimization Specialist" → `performance-optimization-specialist`

## Step 4: Invoke Meta-Agent

Spawn the Meta-Agent to generate the agent definition:

```
Task(subagent_type="general-purpose", description="Generate employee agent")
```

Provide the Meta-Agent with:

1. **Role description** from user
2. **Context**: This is an employee for the company organization
3. **Operating Mode**: `$OPERATING_MODE` (determines where files are created)
4. **Instructions**:
   - Read `.claude/agents/meta-agent.md` for generation rules
   - Read existing company agents for patterns:
     - Multi-project mode: Check `$COMPANY_DIR/employees/[department]/` for patterns
     - Single-project mode: Check `.claude/agents/company/` for patterns
   - Generate an employee-appropriate agent definition
   - The agent should have:
     - Clear capabilities section
     - Structured output format
     - Rules for working within the organization hierarchy
     - Self-validation checklist

5. **Employee-Type Requirements**:
   - If `--type=persistent`: Long-running employee with deep context accumulation
   - If `--type=consultant`: On-demand specialist for specific engagements
   - Should report to department's team lead or head
   - Should follow existing patterns in the codebase
   - Should produce structured output compatible with org reporting

## Step 5: Create Employee Files

The file structure differs based on operating mode:

### Multi-Project Mode ($OPERATING_MODE = "MULTI_PROJECT")

Create the employee definition file directly in the employees directory:

1. **Employee File Location**: `$COMPANY_DIR/employees/[department]/[employee-id].md`
2. **Copy from template**: `$COMPANY_DIR/employees/TEMPLATE.md`
3. **Initialize the employee file** with:
   - YAML frontmatter with id, name, type, department, team, status, created
   - Agent definition section with role, capabilities, specializations
   - Memory section initialized for "unassigned" state
   - Empty assignment history

**Employee File Structure** (based on `TEMPLATE.md`):
```yaml
---
id: [employee-id]
name: [Display Name]
type: [persistent | consultant]
department: [department-id]
team: null
status: available
created: [YYYY-MM-DD]
---
```

Initialize memory sections:
- **Active Focus**: "[Role description] - awaiting project assignment"
- **Project**: "unassigned"
- **Last Updated**: current date
- **Context Notes**: "Newly hired employee, awaiting first project assignment"

### Single-Project Mode ($OPERATING_MODE = "SINGLE_PROJECT")

Maintain backward compatibility with v1.1 behavior:

1. Create agent definition: `.claude/agents/company/[department]/[employee-id].md`
2. Create memory directory: `.company/agents/[employee-id]/`
3. Copy templates:
   - `.company/agents/TEMPLATE/memory.md` → `.company/agents/[employee-id]/memory.md`
   - `.company/agents/TEMPLATE/learnings.md` → `.company/agents/[employee-id]/learnings.md`

4. Initialize memory.md with agent context:
   - Set **Active Focus** to the role description
   - Set **Project** to "Available"
   - Set **Last Updated** to current date

## Step 6: Update org.json

Read current `$COMPANY_DIR/org.json` and add the new employee:

1. Add entry to the `employees` array:

```json
{
  "id": "[employee-id]",
  "name": "[Display Name]",
  "type": "[persistent | consultant]",
  "department": "[department-id]",
  "team": "[team-id or null]",
  "status": "available",
  "capabilities": ["[skill1]", "[skill2]", "..."],
  "memoryPath": "[path to employee file or memory directory]",
  "projectAssignments": [],
  "currentProject": null,
  "hireDate": "[ISO 8601 timestamp]"
}
```

**Memory path differs by mode:**
- Multi-project: `".company/employees/[department]/[employee-id].md"`
- Single-project: `".company/agents/[employee-id]/memory.md"`

2. If team is specified and exists, add employee ID to that team's `members` array.

3. Write updated org.json back to file.

**Note for backward compatibility:** If org.json has an `agents` array but no `employees` array, add to `agents` instead (v1.x schema).

## Error Handling

### Agent Definition Generation Failed

If the Meta-Agent fails to create the agent definition:

```
## Agent Generation Failed

The Meta-Agent could not generate an agent definition.

**Error:** [error details]

**Suggestions:**
1. Try a more specific role description
2. Check that `.claude/agents/meta-agent.md` exists
3. Review existing agent patterns in `.claude/agents/company/`

Run again with: /company-hire [more specific description]
```

### Department Not Found

If specified department doesn't exist in org.json:

```
## Department Not Found

The department "[department-id]" does not exist.

**Available departments:**
- engineering
- product
- design

Either:
1. Use an existing department: `/company-hire [role] --department=engineering`
2. Create the department first via org.json
```

### Team Not Found

If specified team doesn't exist in the department:

```
## Team Not Found

The team "[team-id]" does not exist in department "[department]".

**Available teams in [department]:**
- [team-1]
- [team-2]

Either:
1. Use an existing team: `/company-hire [role] --department=[dept] --team=[team]`
2. Omit --team to hire without team assignment
```

## Rules

1. **Always determine operating mode first.** Use `company_resolver` to check for multi-project mode before any other operations.

2. **Always validate company structure.** The company directory must exist with valid org.json.

3. **Generate unique employee IDs.** If an employee ID already exists, append a numeric suffix (e.g., `performance-specialist-2`).

4. **Use Meta-Agent for definition generation.** Do not hardcode agent definitions. The Meta-Agent ensures consistency with existing patterns.

5. **Initialize memory from templates.**
   - Multi-project: Use `$COMPANY_DIR/employees/TEMPLATE.md`
   - Single-project: Use `.company/agents/TEMPLATE/memory.md` and `learnings.md`

6. **Update org.json atomically.** Read, modify, and write org.json as a single operation. Validate JSON before writing.

7. **Employees default to "available" status.** They are ready for assignment immediately after hiring.

8. **Multi-project employees start "unassigned".** `projectAssignments: []` and `currentProject: null`. They must be assigned to a project via `/company-assign` before they can work.

9. **Preserve existing org.json data.** Only add/modify the new employee entry. Never remove or alter existing entries.

10. **Create all files before declaring success.** If any file creation fails, report the error and do not update org.json.

11. **Backward compatibility.** In single-project mode, maintain v1.1 behavior exactly. Users without multi-project needs should see no change.

12. **Always suggest `/company-assign` in multi-project mode.** New employees need project assignments to be useful.
